1) Главная ось архитектуры: детерминированный tick + reducers
1.1 Tick-loop

Серверная симуляция работает с фиксированным шагом 50 мс (20 Hz).

Всё, что меняет состояние мира, проходит через reducers.

В reducers избегаем float (плавающих чисел): только целые (fixed-point / целочисленная математика), иначе детерминизм развалится на разных платформах/компиляторах.

Практика:

Координаты игрока: x, y в fixed-point (например, клетка = 1000 единиц, тогда 1.234 клетки = 1234).

Дистанции для радиуса 5 клеток: сравнение квадратов расстояний, тоже целочисленно.

1.2 Очередь событий на тик

Чтобы детерминизм был железный, входящие действия не исполняем “сразу”, а кладём в таблицу PendingActions и на тик применяем:

сортировка по tick, затем player_id, затем seq

применение в строгом порядке

Это ещё и частично решает rate limiting (ограничение частоты).

2) Модель данных в SpacetimeDB: минимальный набор таблиц

Ниже — таблицы, которые реально понадобятся, без “воды”:

2.1 Мир

WorldState: current_tick, seed, параметры баланса (capacity линии, константы формулы лимита, и т.п.)

Obstacle: непроходимые зоны (для pathfinding (поиск пути) на клиенте или сервере)

2.2 Игроки

Player: позиция (fixed), скорость, энергия (capacity/current/recharge/output), stealth-стат, root_id (если есть), кулдауны

PlayerSession: анти-спам / seq / последний тик активности

2.3 Генераторы и владение

Generator: id, pos, owner_player_id | null, state (neutral/controlled/isolated), production_rate, expires_at_tick

Root можно хранить как поле в Player, но удобнее отдельной сущностью: RootNode (привязка root к генератору + таймер переноса)

2.4 Линии (рёбра графа)

Line: id, a_gen_id, b_gen_id, owner_player_id, capacity, temp, active, cooldown_until_tick

LineRuntime (опционально): кэш расчётов на текущий тик — flow, utilization, parent_edge_in_routing и т.п.

2.5 Активные действия (overcharge/drain и т.п.)

ActiveChannel: player_id, line_id, mode (overcharge/drain), power_per_sec, start_tick, last_apply_tick

PendingAction: входящие команды от клиента (build line, start channel, stop channel, neutralize и т.д.)

3) Самое важное: как считать flow в графе с циклами (детерминированно)

Ты хочешь “самый короткий / самый выгодный путь”. Это опасная зона, если не зафиксировать ровно один алгоритм выбора маршрута, иначе разные клиенты/серверные ветки будут трактовать “выгодность” по-разному.

Я бы зафиксировал так:

3.1 Routing (маршрутизация) = один выбранный путь до Root для каждого генератора

Для каждого игрока на событии “топология изменилась / атака началась/закончилась” выполняем:

Шаг A: построить подграф
Берём только:

генераторы, которые принадлежат игроку и “в сети”

линии игрока, которые active=true

Шаг B: Dijkstra (Дейкстра) от Root
Вес ребра — целочисленный:

базово weight = 1

можно добавить “штраф за горячие линии”: weight += temp / T_DIV

можно добавить “штраф за перегруженность”: weight += (predicted_flow > capacity ? BIG : 0)

Критично для детерминизма: tie-break (разрешение равенств)

если расстояние одинаковое, выбираем ребро с меньшим line_id, затем меньшим gen_id.
Это убирает “рандом” в равных маршрутах.

Результат: для каждого генератора выбирается parent[gen] — куда он “течёт” к Root.

3.2 Aggregation (агрегация flow) снизу вверх

Когда у тебя есть parent[gen], получаешь дерево маршрутизации (на самом деле “лес, стянутый в Root”).

Дальше:

gen_flow = production_rate (и плюс/минус эффекты drain/overcharge)

суммируешь subtree_flow в родителя по parent в топологическом порядке (порядок тоже должен быть детерминирован: сортируй генераторы по расстоянию убыванию, при равенстве по id).

3.3 Когда пересчитывать

Ты предложил “при изменении топологии и атаке” — ок.

Практически:

Любое изменение линий (build/destroy/disable/enable) ставит флаг network_dirty[player] = true

start/stop overcharge/drain тоже ставит dirty, потому что меняется “стоимость”/перегрев и (если ты так хочешь) “выгодность пути”.

4) Температура “с инерцией” и каскады перегрева

На каждом tick (20 Hz) для каждой линии, которая участвует в маршрутизации:

util = flow - capacity

если util > 0: temp += util * HEAT_K

иначе: temp -= COOL_K (но не ниже 0)

Если temp >= 100:

active=false

cooldown_until_tick = current_tick + COOLDOWN_MIN_TICKS (или до остывания до 0 — но тогда тоже через тик пересчитывать)

Каскад получается автоматически, потому что отключение ребра => network_dirty=true => пересчёт маршрутов => перераспределение flow.

5) Overcharge / Drain как “каналы”, применяемые каждый тик

Модель, которая хорошо ложится на tick:

ActiveChannel хранит режим и мощность.

На каждом tick:

проверяем радиус до линии (евклидово, fixed-point)

списываем энергию игрока: cost = power_per_sec / 20

вносим эффект в сеть:

overcharge: добавляет +power в ветку (как “инъекция” в поток) и тем самым повышает flow на маршруте к Root

drain: уменьшает flow (и если flow становится отрицательным — списывает из пула защищающегося, как ты описал)

Тут важно: “куда именно воздействует канал” в графе.
Самый детерминированный вариант:

канал привязан к конкретной line_id

эффект применяется на ребро и распространяется “к Root” по текущему parent-routing (то есть через выбранный маршрут)

6) Стелс “меняет частоту апдейтов” — как сделать в SpacetimeDB

Сервер authoritative, но ты хочешь не всегда показывать точное положение.

Модель:

В таблице Player держим истинную позицию всегда.

В таблице PlayerVisibility для каждого наблюдателя храним “последнюю опубликованную позицию” цели и next_reveal_tick.

На тик:

если цель в стелсе, обновляем “паблик позицию” раз в N тиков (например, 10 тиков = раз в 0.5с)

если нет стелса — раз в тик

Клиент подписывается (subscription — подписка) не на Player, а на PlayerPublicState, где лежит уже “разрешённая” позиция.

7) Клиент 60 FPS при сервере 20 Hz

Чтобы было “как масло”:

Клиент рендерит 60 FPS.

Состояние мира приходит 20 раз/сек.

Между апдейтами делаешь interpolation (интерполяция):

игроки: интерполяция позиции по времени

линии/температура: тоже можно линейно сглаживать визуализацию, но серверные значения не трогать

Для рендера “нативно и быстро” на браузере:

2D: PixiJS (ПиксиДжиЭс) на WebGL (веб-графика) — типичный выбор

или чистый Canvas 2D, но на MMO-карте с кучей линий может быть тяжелее

8) Rate limiting без сложной системы

Для MVP хватит простого:

в PlayerSession хранить action_budget на тик, например 5 “очков”

каждая команда стоит 1–3 очка

бюджет восстанавливается каждый тик

Это сразу уберёт спам build/destroy/start/stop.

9) Что я бы зафиксировал “по умолчанию” (чтобы не тормозить разработку)

Одна линия = ребро между двумя генераторами (как ты сказал), пересечения на карте — purely visual (чисто визуально), в логике не взаимодействуют.

Routing — Dijkstra по активным линиям с детерминированным tie-break.

Любой перегрев/отключение ребра делает network_dirty.

Пересчёт сети делаем не каждый тик, а только когда dirty.

Все числа — целые, в fixed-point.
